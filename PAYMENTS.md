# Платежи Pithy Pro — как работает и что делать при подключении кассы

## Что такое Pro

Подписка на 399 ₽/месяц. Даёт:
- Безлимит энергии (уроки без ожидания)
- Золотой значок PRO в рейтинге
- Копилка слов без лимита (бесплатно видно первые 20 слов)

Касса — **ЮKassa**. Код готов целиком; пока ключи кассы не заданы, кнопка оплаты работает в стаб-режиме: показывает «оплата откроется совсем скоро».

## Как проходит платёж (цепочка)

1. Пользователь жмёт **«Оформить — 399 ₽/мес»** на экране Pro (открывается из профиля, из экрана «Энергия закончилась» или после первого прохождения Финала модуля).

2. Приложение вызывает edge-функцию `create-payment` (с токеном пользователя). Она:
   - Создаёт запись в таблице `payments` (статус `pending`)
   - Создаёт платёж в ЮKassa
   - Возвращает ссылку на страницу оплаты

3. Пользователь оплачивает на странице ЮKassa и возвращается в приложение.

4. ЮKassa присылает уведомление `payment.succeeded` на функцию `payment-webhook`. Функция **не верит присланным данным**: перезапрашивает платёж напрямую у API ЮKassa по секретным ключам и действует только по этому ответу. Повторное уведомление подписку второй раз не продлит.

5. Вебхук помечает платёж `succeeded` и вызывает SQL-функцию `activate_subscription`:
   - +30 дней к подписке (досрочная оплата дни не сжигает)
   - У пользователя ставится `has_subscription = true` и `subscription_until = дата окончания`

6. Истечение проверяется лениво при старте урока (SQL-функция `expire_subscription`). Подписка, выданная вручную без даты (`subscription_until` пустой), не истекает никогда.

## Где какой код

| Файл | Роль |
|------|------|
| `src/features/pro/ProPaywall.jsx` | Экран подписки с кнопкой оплаты |
| `src/shared/api/subscriptionApi.js` | Вызов create-payment из приложения |
| `supabase/functions/create-payment/index.ts` | Создание платежа в ЮKassa (или стаб) |
| `supabase/functions/payment-webhook/index.ts` | Приём уведомления, активация подписки |
| `supabase_schema.sql` | Таблица `payments`, колонка `subscription_until`, функции `activate_subscription` / `expire_subscription` |
| `supabase/config.toml` | Настройки деплоя функций (verify_jwt уже прописан) |

## Чек-лист: подключение кассы

Шаги по порядку:

**1. Применить SQL**
Откройте Supabase → SQL Editor и выполните конец файла `supabase_schema.sql` — секции «Энергия v3» и «Подписка Pithy Pro» (если ещё не применяли).

**2. Привязать CLI** (один раз с этой машины)
```bash
supabase login
supabase link --project-ref <ref>
```
`<ref>` — из адреса дашборда: `https://supabase.com/dashboard/project/<ref>`

**3. Задеплоить функции** (из корня проекта)
```bash
supabase functions deploy create-payment
supabase functions deploy payment-webhook
```
Флаг `--no-verify-jwt` для вебхука указывать **не нужно** — настройка `verify_jwt = false` уже закреплена в `supabase/config.toml`, CLI применит её сам.

**4. Задать секреты** (ключи из личного кабинета ЮKassa)
```bash
supabase secrets set YOOKASSA_SHOP_ID=xxxxx YOOKASSA_SECRET_KEY=live_xxxxx
```
Или через дашборд: Edge Functions → Secrets.

Опционально: `PAYMENT_RETURN_URL` — куда вернуть пользователя после оплаты (по умолчанию `https://pithy.app/`).

**5. Настроить вебхук в ЮKassa**
Интеграция → HTTP-уведомления → указать URL `https://<ref>.supabase.co/functions/v1/payment-webhook`, отметить событие `payment.succeeded`.

**6. Проверить**
Тестовая оплата → в логах функции должно появиться `[PAY] подписка активирована`, в профиле должно появиться «Pro активна до …».

## Почему у вебхука отключена проверка JWT

- По умолчанию Supabase пускает к функции только запросы с токеном залогиненного пользователя.
- `create-payment` вызывает пользователь из приложения — токен есть, проверка включена.
- `payment-webhook` вызывает сервер ЮKassa — токена у него нет, поэтому входная проверка отключена.

Это не дыра: защита внутри функции (перепроверка платежа у API кассы по секретным ключам).

## Передеплой

- Нужен только если менялся **код функции**: `supabase functions deploy <имя-функции>`.
- Правки SQL применяются через SQL Editor, деплой функций для этого не нужен.
- Правки клиентского кода (ProPaywall и т.п.) деплоятся как обычно через Vercel, к функциям отношения не имеют.

## Если что-то не работает

| Симптом | Причина и решение |
|---------|-------------------|
| Кнопка говорит «оплата скоро» | Секреты `YOOKASSA_*` не заданы — задать и передеплоить не нужно, подхватятся сами |
| Оплата прошла, но Pro не появился | Проверить: URL вебхука в ЛК ЮKassa; логи в Dashboard → Edge Functions → payment-webhook → Logs (искать `[PAY]`) |
| Вебхук получает 401 | Функцию задеплоили с включённой проверкой JWT — передеплоить (verify_jwt=false в config.toml должен подхватиться) |
| Значок PRO не виден в рейтинге | SQL-секция «Подписка Pithy Pro» не применена (там обновляется get_leaderboard) |
