# Cloudflare: домен + CDN для видео — пошаговая настройка

Отложенная задача. Выполняется руками в кабинетах Cloudflare/Supabase,
кода не требует. Можно делать в любой момент — пока не сделано, приложение
работает как обычно, просто без edge-кэша (третьего слоя скорости видео).

## Зачем это нужно

Видео ленты лежат в Cloudflare R2 (бакет `pithy-files`) и сейчас отдаются
по публичному адресу без CDN-кэша. Подключение собственного домена даёт:

1. **Edge-кэш**: файл после первого запроса живёт на серверах Cloudflare по
   всему миру — пользователь качает с ближайшего, а не из бакета.
2. **HTTP/3 + 0-RTT**: экономия ~1-2 сек на установке соединения (наш тест
   на 3G показывал ~2.2 сек на старт каждого запроса).
3. Красивые ссылки для репостов и брендинг (`pithyapp.com` вместо
   `pithy-app.vercel.app`).
4. Исходящий трафик R2 **бесплатный** — раздача видео не стоит ничего.

Стоимость: домен ~$10-12/год, тариф Cloudflare Free — $0.

Полная скорость видео = три слоя: (1) файл оптимизирован
`tools/prepare-video.bat` — уже работает; (2) префетч соседних слайдов —
уже в коде ленты; (3) CDN — этот файл.

## Шаг 0. Домен в аккаунте Cloudflare

Custom domain для R2 работает только с доменом, который обслуживается
(«зона») в том же аккаунте Cloudflare, что и бакет.

### Путь А — купить домен в Cloudflare (проще, нужна зарубежная карта)

1. [dash.cloudflare.com](https://dash.cloudflare.com) → войти в аккаунт, где живёт R2.
2. Слева: **Domain Registration → Register Domains**.
3. Ввести желаемое имя (например `pithyapp`) → выбрать свободный вариант
   (`.com` ≈ $10-11/год; цены по себестоимости, WHOIS-privacy бесплатно).
4. **Purchase** → оплата. Принимаются Visa/MC/PayPal; российские карты НЕ
   проходят — тогда Путь Б. Домены `.ru` Cloudflare не продаёт.
5. Готово: зона создаётся сама, nameservers уже cloudflare'овские.

### Путь Б — домен у другого регистратора (reg.ru, Timeweb, Namecheap...)

1. Купить/взять существующий домен у регистратора (`.ru` тоже подходит).
2. Cloudflare → **Websites → Add a site** → ввести домен без `www` → Continue.
3. Выбрать план **Free** (его хватает на всё: R2 domain, Cache Rules, HTTP/3).
4. Cloudflare импортирует текущие DNS-записи → Continue.
5. Показаны **два nameserver'а** вида `alice.ns.cloudflare.com` /
   `bob.ns.cloudflare.com` — скопировать.
6. В кабинете регистратора → настройки домена → «DNS-серверы / Nameservers»
   → заменить на эти два → сохранить.
7. Ждать статус **Active** в Cloudflare (обычно 15 мин — 2 ч, максимум 24 ч;
   придёт письмо).

### Рекомендуемая раскладка

| Адрес | Куда указывает |
|---|---|
| `pithyapp.com` (корень) | приложение на Vercel (Vercel → Settings → Domains, он подскажет DNS-запись) |
| `media.pithyapp.com` | бакет R2 — настраивается в Шаге 1 |

## Шаг 1. Подключить поддомен к бакету

1. Dashboard → **R2 Object Storage** → бакет **pithy-files**.
2. Вкладка **Settings** → блок **Public access** → **Custom Domains** →
   **Connect Domain**.
3. Ввести `media.<домен>` → Continue → Connect. DNS-запись создастся сама.
4. Дождаться статуса **Active** (минуты).

## Шаг 2. Cache Rule — кэшировать файлы на edge

1. Dashboard → **Websites** → выбрать домен → **Rules** → **Create rule** →
   **Cache Rule**.
2. Имя: `media-cache`.
3. Условие: **Hostname** equals `media.<домен>`.
4. Then: **Cache eligibility → Eligible for cache**.
5. **Edge TTL** → Override origin → **1 year**;
   **Browser TTL** → Override origin → **1 year**.
   Это безопасно: имена файлов уникальны, новое видео = новое имя.
6. **Deploy**.

## Шаг 3. Протоколы

Домен → слева **Network**:
- **HTTP/3 (with QUIC)** → ON;
- **0-RTT Connection Resumption** → ON.

## Шаг 4. Переключить приложение на новый домен

Ссылки строятся из секрета `R2_PUBLIC_BASE` edge-функции — код менять не надо.

1. **Сначала запиши старое значение**: Supabase Dashboard → Edge Functions →
   Secrets → `R2_PUBLIC_BASE` (понадобится для миграции ниже).
2. Поменяй `R2_PUBLIC_BASE` на `https://media.<домен>` (без слэша в конце).
   Новые загрузки сразу пойдут с новым доменом.
3. Старые ссылки в БД мигрируются одним скриптом в SQL Editor
   (подставь СТАРУЮ базу из п.1 и НОВУЮ; перед запуском можно сделать
   бэкап: Database → Backups):

```sql
-- files: реестр файлов админки
update public.files
set r2_url = replace(r2_url, 'https://СТАРАЯ_БАЗА', 'https://media.ДОМЕН');

-- curricula: видео и постеры ленты
update public.curricula
set video_url  = replace(video_url,  'https://СТАРАЯ_БАЗА', 'https://media.ДОМЕН'),
    poster_url = replace(poster_url, 'https://СТАРАЯ_БАЗА', 'https://media.ДОМЕН')
where video_url is not null or poster_url is not null;

-- lessons.script: внутри графов уроков тоже лежат абсолютные ссылки
update public.lessons
set script = replace(script::text, 'https://СТАРАЯ_БАЗА', 'https://media.ДОМЕН')::jsonb;
```

(Можно попросить Клода сгенерировать скрипт с уже подставленными адресами.)

4. CORS бакета не трогаем: загрузки (PUT) идут по presigned-ссылке напрямую
   в R2, а правила CORS бакета действуют и для нового домена.

## Шаг 5. Проверка

1. Открыть `https://media.<домен>/<любой файл>` → грузится.
2. DevTools → Network → заголовки ответа: первый запрос
   `cf-cache-status: MISS`, после обновления страницы — **HIT** (кэш работает);
   есть `alt-svc: h3=...` — HTTP/3 включён.
3. В приложении: лента играет видео, плеер урока качает файлы
   (предзагрузка — первый кандидат заметить проблемы CORS),
   загрузка нового файла в админке кладёт ссылку на новый домен.

## Можно ли отложить? Да

Пока шаги не выполнены, всё работает на текущих ссылках. Отсрочка лишь
накапливает старые ссылки в БД, но миграции (Шаг 4.3) всё равно, сколько их.
Разумный дедлайн — до публичного запуска, чтобы у пользователей видео
летало с первого дня.
