# Автотестирование Pithy — план реализации + система ловли багов

Этот файл — самостоятельное ТЗ для исполнителя (младшая модель). Цель:
построить **автоматические E2E-тесты**, которые имитируют живого
пользователя (скроллит ленту, проходит уроки, тратит энергию и билеты),
и **систему ловли багов** — каждый тест автоматически падает при ЛЮБОЙ
ошибке консоли или сети, с артефактами (скриншот, видео, трейс действий).
Концепция приложения — в `PROJECT.md`.

Выполнять этапы A → F по порядку, каждый — отдельная сессия. Каждый этап
завершён, только когда его тесты зелёные локально.

## Правила для исполнителя

1. Правила проекта из `CLAUDE.md` обязательны: русский язык; поднять
   `APP_VERSION` при изменении кода приложения (изменения только в `e2e/`
   версию не поднимают); новые файлы — строкой в `STRUCTURE.md`; лимит
   250/400 строк на файл (на тест-файлы тоже); в конце ответа — список
   файлов с итоговым числом строк; в git пушить только по явной просьбе.
2. **Секреты**: тест-аккаунты спросить у пользователя (обычный + админ,
   лучше завести отдельные «e2e-юзер»/«e2e-админ»). Логин/пароль хранить в
   `.env.test` (добавить в `.gitignore` ПЕРВЫМ делом, проверить `git status`
   что файл не отслеживается). В код и в репозиторий пароли не попадают.
3. **База данных общая с продом** (отдельного тест-бэкенда нет): тесты
   гонять только на локальном dev-сервере, писать только под тест-аккаунтами
   и только в спец-модуль «E2E-ТЕСТ» (этап C). Чужие данные не трогать.
   Голосования/лайки от тест-аккаунта допустимы, но не в цикле.
4. **Не чинить найденные баги в той же сессии** — фиксировать в отчёт
   (формат в конце), чинить отдельно по утверждённому списку.
5. Известные компромиссы v1 — НЕ баги: сервер верит клиенту в
   подсказки/ошибки/время; пушей гонки нет; `npm run lint` даёт ~46 ошибок
   react-hooks (их чинит этап 5 плана стабилизации в PROJECT.md).

## Этап B — система ловли багов (ядро, 1 сессия)

Смысл: сценарий может «пройти», но если в консоли была ошибка — это баг,
и тест обязан упасть. Делается один раз фикстурой, дальше ВСЕ тесты
наследуют её автоматически.

1. `e2e/fixtures.js` — расширенный `test`:
   ```js
   import { test as base, expect } from '@playwright/test'

   // Шум, который багом не считается — пополнять ТОЛЬКО с комментарием почему
   const IGNORE = [
     /the play\(\) request was interrupted/i, // автоплей-политика браузера
     /401.*auth\/v1/i,                        // гость без сессии — норма
   ]

   export const test = base.extend({
     page: async ({ page }, use) => {
       const bugs = []
       page.on('console', m => {
         if (m.type() === 'error' && !IGNORE.some(r => r.test(m.text())))
           bugs.push(`console: ${m.text()}`)
       })
       page.on('pageerror', e => bugs.push(`pageerror: ${e.message}`))
       page.on('requestfailed', r => {
         if (!IGNORE.some(rx => rx.test(r.url())))
           bugs.push(`requestfailed: ${r.url()} ${r.failure()?.errorText}`)
       })
       page.on('response', r => {
         if (r.status() >= 500) bugs.push(`http ${r.status()}: ${r.url()}`)
       })
       await use(page)
       expect(bugs, 'Ошибки консоли/сети во время теста').toEqual([])
     },
   })
   export { expect }
   ```
2. Все тесты импортируют `test`/`expect` из `./fixtures.js`, не из
   `@playwright/test` напрямую.
3. Хелпер медленной сети `e2e/helpers/network.js`: включение эмуляции 3G
   через CDP (`client.send('Network.emulateNetworkConditions', ...)`) —
   понадобится этапам C и E.
**Готово, когда**: тест, в котором страница специально кидает ошибку
(`page.evaluate(() => { throw new Error('test') })` не подходит — он
падает сам; проверить через `page.evaluate(() => console.error('x'))`),
падает с текстом «Ошибки консоли/сети».

**✅ ВЫПОЛНЕНО 2026-07-18.** `e2e/fixtures.js` (расширенный test, валит при
ошибке консоли/сети) + `e2e/helpers/network.js` (3G через CDP). Проверено
селф-тестом: нарочная `console.error` роняет тест, а транспортный шум сети
(сбросы к Supabase/Google Fonts на сети РФ) — игнорируется. IGNORE-список
строго транспортный (ERR_CONNECTION_RESET/TIMED_OUT, Failed to fetch, font-CDN,
автоплей, 401 гостя); настоящие баги ловятся: JS-исключения (pageerror),
HTTP 5xx, битые ассеты (404-текст в консоли по URL).

## Этап C — гостевые сценарии + тест-модуль (1–2 сессии)

Гость не пишет на сервер (кроме событий анализа) — эти тесты безопасны
и должны стать самым толстым слоем.

1. **Попросить пользователя (или через e2e-админа) создать модуль
   «E2E-ТЕСТ»**: опубликованный, с видео, структура «Старт → 1 урок →
   Финал», внутри урока — текстовая нода + «выбери слово» (2 варианта) +
   про-текст с переводом. Его id записать в `e2e/config.js` — все
   сценарии ходят в него, чужой контент не трогают.
2. `data-testid` в коде приложения (это изменение кода → поднять версию):
   расставить на ключевые элементы, которые сейчас ищутся по классам/тексту:
   слайд ленты, кнопка «Изучить фразу», нода урока на схеме, вариант ответа
   (с признаком correct в testid нельзя — подсказка чит-скриптам; вместо
   этого в тест-модуле правильный вариант всегда первый), кнопка закрытия
   итогов, значки HUD. Список дополнять по мере написания тестов.
3. Тесты `e2e/guest.spec.js`:
   - лента загрузилась, скролл на 3 слайда вниз и обратно — слайды сменяются;
   - тап «Включить звук» — оверлей исчез;
   - тап по лайку гостем → открылась форма входа;
   - «Изучить фразу» тест-модуля → схема: Старт активен, урок под замком;
   - пройти Старт → урок разблокировался;
   - пройти урок: ответить на «выбери слово», дождаться итогов, увидеть
     звёзды в итогах, закрыть → на схеме у урока звёзды;
   - deep-link `/?m=<id тест-модуля>` → первый слайд = тест-модуль.
4. Тот же прогон с эмуляцией 3G (хелпер этапа B) — тайминги в тестах
   держать щедрыми (`expect(...).toBeVisible({ timeout: 30_000 })`).
**Готово, когда**: гостевой пакет зелёный подряд 3 запуска (стабильность!),
и падает, если специально сломать что-то в коде (проверить один раз).

## Этап D — сценарии залогиненного (тест-аккаунт, 1–2 сессии)

1. Логин один раз в `e2e/auth.setup.js` через форму входа (креды из
   `process.env`), сохранить `storageState` в `test-results/`; остальные
   тесты стартуют уже залогиненными (project dependencies Playwright).
2. Сброс состояния: перед пакетом e2e-админ жмёт ⟲ на схеме тест-модуля
   (отдельный setup-тест под админ-кредами) — прогресс/XP тест-юзера по
   тест-модулю обнуляется. Билеты НЕ сбрасываются (module_tickets навсегда)
   — тесты билетов писать так, чтобы работали в обоих случаях: «билет
   получен» ИЛИ «уже был получен раньше» — оба валидны, важно что счётчик
   ведёт себя правильно.
3. Тесты `e2e/user.spec.js`:
   - после входа HUD виден: уровень, 🎟, ⚡; тап по каждому — своё мини-окно;
   - старт НОВОГО урока списывает 1 ⚡ (сравнить текст значка до/после);
   - пересдача пройденного и Старт/Финал — ⚡ не меняется;
   - Финал: панель подсказок видна; раскрыть перевод → 1 огонёк погас;
     раскрыть ТО ЖЕ сообщение снова → второй огонёк НЕ погас;
   - пройти Финал с ≤3 подсказками → в итогах блок билета («получен» или
     «уже был получен»), счётчик 🎟 не уменьшился;
   - звёзды: пройти урок с ошибкой (кликнуть неверный вариант, потом
     верный) → в итогах 2★; на схеме — лучший результат.
4. Тесты `e2e/race.spec.js` (если в БД есть активная/будущая гонка —
   иначе `test.skip` с причиной): баннер в Рейтинге со строкой про билет;
   тап → попап; проверка веток попапа по текущему состоянию аккаунта
   (без билета — нет кнопки; с билетом — «Открыть доступ» и счётчик −1
   после тапа; повторный тап баннера → «уже оплачен», счётчик не меняется).
