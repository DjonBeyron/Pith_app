## Задача: примитивная админ-панель (этап 1)

Это самый первый этап разработки приложения. Цель этапа — добиться **стабильной** загрузки
файлов на сервер и их получения пользователем как на ПК, так и на смартфоне. Никакого
конструктора уроков на этом этапе нет — только файлы.

Два таба: **Пользователь** и **Админ**.

### Таб «Пользователь»
- Видит только файлы, которые есть на сервере.
- Может воспроизвести: фото (просмотр), видео (воспроизведение), аудио (воспроизведение).

### Таб «Админ»
- Создаёт строку → выбирает файл с ПК.
- В строке видно: выбранный файл, вес файла (столбец), статус (на сервере / нет), время
  загрузки на сервер.
- Может удалить файл с сервера прямо из строки.
- **Все изменения применяются только после нажатия главной кнопки «Синхронизировать с
  сервером»** — выбор файла и удаление до этого момента — локальное, неподтверждённое действие.
- Кнопка «Активировать дебаг» — включает подробные логи в консоли: как файл уходит от админки,
  как файл загружается у пользователя, время загрузки и т.д.

## Архитектурное решение на будущее: предзагрузка во время пауз сценария (2026-06-16)

В чат-сценарии уже есть искусственные паузы между сообщениями (задержки для реалистичности
диалога — «вы присоединились» → пауза → «учитель присоединился» и т.п., см. триггеры
`after_delay`/`startDelay` в старом проекте). Идея: использовать эти и так существующие паузы
как окно для фоновой подгрузки медиафайлов из следующих 1–3 сообщений вперёд — пока пользователь
читает текущее сообщение, следующие файлы тихо скачиваются и кэшируются. К моменту, когда
сценарий доходит до них, показ происходит мгновенно, без видимого лага.

**Не проверено:** сколько файлов реально успевает подгрузиться за типичную паузу (3–5 сек) на
настоящем 3G — наш тест замерял только скорость маленького JSON-запроса (пара КБ), не скорость
скачивания самого файла (фото/видео — сотни КБ — несколько МБ). Нужен отдельный тест: скачать
реальный файл из R2 на 3G и замерить КБ/сек, чтобы рассчитать, сколько секунд паузы нужно на
сколько файлов вперёд.

### Финальная схема (согласовано 2026-06-16)

Тест на реальном 3G показал: каждый новый запрос «стоит» ~2.2 сек на установку соединения почти
независимо от размера файла (32 КБ и 1.9 МБ грузились за одинаковое время) — то есть упираемся
не столько в скорость канала, сколько в задержку на старте каждого файла.

Прямое использование текущего механизма (полная докачка в `Blob`, строго по очереди) для
финального чата опасно тремя вещами: (1) память — блобы копятся вечно; (2) если очередь не
успела — пользователь видит пустую заглушку вместо файла; (3) один зависший файл блокирует всю
очередь позади себя. Договорились так:

- **Фоновая предзагрузка** (текущий механизм: очередь + ретраи) — только для сообщений,
  которые ещё не показаны, заранее, во время пауз сценария.
- **Текущее показанное сообщение**, если для него ещё не готов blob — не ждать его, а сразу
  отдать нативный `<video>/<audio> src=...>` напрямую (браузер сам стримит с буферизацией).
  Хуже, чем мгновенный blob, но не зависание.
- **Лимит памяти: держим в памяти blob'ы только последних 5 сообщений** (скользящее окно),
  более старые — выгружаем (`URL.revokeObjectURL` + убираем из state). Важно для слабых
  Android-телефонов с небольшим объёмом RAM — лесон с 20+ медиафайлами по 2-3 МБ не должен
  копиться в памяти целиком.
- **Зависший файл не должен стопорить очередь навсегда** — нужен потолок ожидания, после
  которого очередь идёт дальше, а зависший файл продолжает попытки в фоне отдельно.

**Когда делать:** часть следующего этапа (чат-плеер), не текущей админ-панели.

### Реализовано в табе «Пользователь» как прототип/тест (2026-06-16)

Уточнения к схеме выше, добавленные после теста и реально закодированные (не только описаны):

- **Фото никогда не выгружаются** — пустое фото в чате выглядело бы как явная дыра, а фото
  занимают не так много памяти, как видео. Лимит буфера (5) считается только для видео/аудио.
- **Видео при выгрузке не пропадает совсем** — перед удалением тяжёлого blob'а захватывается
  один кадр (через скрытый `<video>` + `<canvas>`) и остаётся как стоп-кадр вместо пустоты.
- **Буфер на 5 видео/аудио, вытеснение — по дистанции от текущей позиции в чате**, не по
  времени загрузки. Пример: в буфере 20-24 (текущая позиция ~24), пользователь подгружает
  №10 по требованию → вытесняется №20 (он дальше всех от 24). Дальше пользователь подгружает
  №2 → должен вытесниться №10 (он теперь дальше всех — 22 шага от текущей позиции против 3-4
  у 21-24), а не №21 просто потому что он «дольше всех висит» (первая версия буфера ошибочно
  вытесняла именно по этому принципу — поправлено).
- **Подгрузка по требованию**: клик на выгруженный файл (стоп-кадр видео или плейсхолдер
  аудио) запускает повторную загрузку этого конкретного файла через тот же механизм
  (очередь/повтор); попав в буфер, такой файл сам станет первым кандидатом на вытеснение,
  если позиция в чате не сдвинется к нему.

**Найденные и исправленные ошибки (тест на Android, 2026-06-16):**
1. Предзагрузка прогоняла весь список файлов почти мгновенно (22 файла за ~17 сек), не
   обращая внимания на то, что показ на экране идёт по одному каждые 3 сек. Буфер на 5
   успевал заполниться и провернуться несколько раз ещё до того, как пользователь реально
   дошёл до этих сообщений. Фикс: предзагрузка теперь не уходит дальше, чем на
   `PREFETCH_LOOKAHEAD = 3` сообщения вперёд от уже показанного (параметр `allowUpTo`).
2. Вытеснение было устроено как FIFO («кто дольше висит в буфере») — из-за этого подгрузка
   старого сообщения по требованию вытесняла случайное соседнее свежее сообщение, а не саму
   себя при следующей подгрузке. Заменено на вытеснение по дистанции от текущей позиции
   (параметр `currentIndex`) — буфер теперь всегда защищает то, что рядом с «живым краем»
   чата, а не то, что просто недавно подгрузилось.

Это пока живёт в табе «Пользователь» как полигон для теста на слабом Android — перенос в
реальный чат-плеер (с нативным стримингом для «текущего» сообщения) — следующий этап.

## Система уроков — концепция (2026-06-17)

### Идея

Урок = цепочка нод на canvas-холсте. Каждая нода — одно сообщение в чате (аудио, фото, видео,
текст). Между нодами — переходы по триггерам: «если [событие] → тогда [следующая нода]».
Граф сохраняется на сервер как JSON. Клиент скачивает его один раз и запускает state machine
локально — серверная логика триггеров не нужна (урок детерминирован).

### Что хранится на сервере

Таблица `lessons` (добавить в `supabase_schema.sql`):
```
id uuid, title text, script jsonb, created_at timestamp
```

`script` — один JSON-документ:
```json
{
  "nodes": [
    {
      "id": "n1",
      "x": 120, "y": 80,
      "type": "audio",
      "file_id": "uuid",
      "triggers": [
        { "if": "played",          "then": "n2" },
        { "if": "timer", "ms": 3000, "then": "n3" }
      ]
    }
  ]
}
```

Файлы (аудио/фото/видео) не дублируются — ноды ссылаются на существующую таблицу `files`.

### Три размера ноды

| Размер | Что видно |
|--------|-----------|
| **nano** | Только порядковый номер ноды. Цвет фона по типу: фото = мягкий зелёный, аудио = мягкий синий, видео = мягкий фиолетовый, текст = нейтральный серый |
| **mini** | Порядковый номер + название типа (из выпадающего меню типа) |
| **max** | Порядковый номер + выпадающее меню типа («Текстовое / Голосовое / Фото / Видео сообщение») + триггеры: «Если» и «Тогда» — кастомные выпадающие меню как в старом `BlockEditorChat.jsx` |

Переключение между размерами — кликом на ноду (nano → mini → max → nano).

### Типы триггеров «Если»

| ID | Описание |
|----|----------|
| `played` | Аудио/видео прослушано до конца |
| `timer` | Прошло N секунд с момента появления ноды |
| `timer_after_play` | Прошло N секунд с момента окончания воспроизведения |
| `photo_shown` | Фото появилось на экране |

«Тогда» — всегда выбор существующей ноды из списка.

### Canvas-страница

Открывается поверх вкладок (простое переключение страниц в `App.jsx`, без роутера).
Верхний левый угол — кнопка-заглушка (будущие настройки урока).
Соединения между нодами — bezier-кривые с органическим wobble (`neuronPath` из старого проекта).
Узлы перетаскиваются. Холст панорамируется.

## Новая задача: Layout-система чата (плеер)

### Принцип

Чат живёт на `flex-direction: column-reverse`. Это даёт:
- Первое сообщение всегда у низа экрана
- Каждое новое сообщение появляется снизу, старые уходят вверх
- Пузырь при росте высоты (перенос текста) расширяется вверх, не вниз
- `scrollTop=0` всегда = виден низ чата, никакого JS `scrollToBottom`

Панели снизу (`ChooseWord`, `PhraseAssembly`) — flex-сиблинги под фидом.
При появлении панели фид сжимается, история автоматически смещается вверх.

Анимация панели: IN — система монтирует + CSS slide-up. OUT — модуль сигналит `onDone()` →
система ставит `exiting` → CSS slide-down → после `transitionEnd` система размонтирует.
Модуль не знает о своей анимации ухода.

```
┌─────────────────┐
│   PlayerTopBar  │  flex-shrink: 0
├─────────────────┤
│                 │
│   PlayerFeed    │  flex: 1 · column-reverse · overflow-y: auto
│   (история ↑)   │
│                 │
│  [последнее]    │  ← всегда у низа фида
├─────────────────┤
│  BottomPanel    │  flex-shrink: 0 · высота авто · slide-up анимация
└─────────────────┘
```

### Шаг 6 — Easy-bounce (лёгкое пружинящее покачивание)

**Что делаем:**
- Заменяем `ease` на кастомный `cubic-bezier` с лёгким перелётом (overshoot):
  - Slide-up: `cubic-bezier(0.34, 1.20, 0.64, 1)` — чуть вылетает вверх и оседает
  - Slide-down: `cubic-bezier(0.36, 0, 0.66, -0.10)` — лёгкий уход с подтяжкой
- Длительность: slide-up ~320ms, slide-down ~260ms (уход быстрее — так ощущается живее)
- **Только `transform` и `opacity`** — GPU-ускорение, нет layout/paint
- `will-change: transform` на `.bottomPanel` — подсказка браузеру заранее

**Важно по оптимизации:**
- Никаких анимаций `height`, `padding`, `margin` — только `transform`
- Чат сжимается не анимацией высоты панели, а тем что панель в потоке (`flex`)
  и появляется мгновенно по высоте (только transform двигает её визуально)
- На слабых телефонах `prefers-reduced-motion` → убираем bounce, оставляем simple ease

**Файлы:** `panels.css`

**Тест:** смотреть на реальном телефоне — покачивание должно быть едва заметным,
не агрессивным. Проверить в Chrome DevTools → Performance что нет layout thrash.

## Пользовательский интерфейс: видео-лента (концепция согласована 2026-07-05)

### Суть
Приложение встречает пользователя вертикальной видео-лентой (как TikTok).
**Одно видео = одна фраза = один модуль** (`curricula`). Уроки внутри модуля —
слова этой фразы. Кнопка «Изучить фразу» на видео ведёт на схему модуля
(Старт → уроки-слова → Финал — уже готова, `ModuleGraph`). Существующее ядро
(плеер, XP, анализ знаний, редактор) не трогаем — меняется только обёртка.

### Навигация
- Нижний бар: **[Уроки]** (лента) · **[Профиль]**; для `is_admin` — третья
  иконка «Админ» (текущие админ-вкладки + canvas-редактор, пользователи её не видят).
- Поверх ленты сверху две вкладки: **(Рекомендации)** | **(Мои уроки)**.
- Рекомендации — все опубликованные модули-видео.
- Мои уроки — начатые модули; два режима: видео-скролл (без HUD справа, снизу

- Гость смотрит ленту без ограничений; может пройти диагностику и первый урок

## Сложность фразы на слух — краудсорсинг (решения 2026-07-09)

### Суть
У каждого видео-фразы в ленте — иконка сложности (три полоски-«сигнал»),
она же кнопка голосования. Тап → панель выезжает от кнопки (scale от 0 +
рост в ширину). Один вопрос — «Понял на слух?»: 🟢 Легко (с первого раза) ·
🟡 Средне (переслушал/частично) · 🔴 Сложно (открыл текст). Чипы-причины
(быстрая речь, незнакомые слова) — НЕ в первой версии.

- Голос перезаписываемый, бессрочный; одна строка на (user, module).

  `curricula.difficulty` + `difficulty_votes` (пересчёт триггером БД при
  каждом голосе — лента/фильтр не агрегируют на лету).
- До ~5 голосов иконка серая («мало оценок») — порог на клиенте
  (`MIN_DIFFICULTY_VOTES`), один голос фразу не красит.
- В панели голосования чужое распределение не показывается — только свой
  выбор (анти-якорение). Тап по кружку применяет голос, тап вне — закрывает.
- В «Моих уроках» иконка показывает **свой голос в приоритете** (иначе общий);
  голосовать можно и там — та же запись в БД.
- Гость: тап по голосу → форма входа (как лайк).

### Неявные сигналы (этап 3, потом)
- Снимок «первой встречи» с фразой: проигрывания + открыл ли текст; после
  первого сеанса замораживается навсегда (повторные просмотры не считаются).
- Учитывается ТОЛЬКО от непроголосовавших, малым весом (~0.3), как паттерн
  (переслушал И открыл текст = сильный сигнал), а не одиночное событие.
- Явный голос всегда важнее: проголосовал — его неявные сигналы выкидываем.

### Фильтр по сложности (этап 2, потом)
- Одна глобальная кнопка: маленькая, справа в шапке; состояние общее для
  «Рекомендаций» и «Моих уроков», запоминается.
- Панель с той же анимацией; мультивыбор 🟢🟡🔴; «все выключены» = «все
  включены».
- Рекомендации: фильтр по общей оценке + подмешивание серых фраз ~1 из 5–7
  (холодный старт новых фраз). «Мои уроки»: фильтр по своему голосу (иначе
  общий), серые видны всегда.

### Этапы
1. **БД + голосование в ленте** (иконка, панель, свой голос) — в работе.
2. Фильтр в шапке.
3. Неявные сигналы первой встречи; позже — взвешивание по уровню ученика.

## Супергонка + вкладка «Рейтинг» + достижения (решения 2026-07-09)

### Супергонка — суть
Еженедельное соревнование по выходным (сб+вс), пройти можно один раз.
Спец-страница гонки: тема (жизненная ситуация — «буйный таксист»,
«доставщик задерживается»...), порог XP для открытия, список заданий
(ссылки на готовые уроки — визуально как «Мои уроки» в режиме списка),
сверху крупный вход в саму гонку.

### Админ-конструктор гонки
- Аналог графа урока, но без Старта и без линий: просто упорядоченный
  список уроков, которые админ выбирает из готовых уроков приложения
  (ссылки, не копии). Порядок можно менять.
- Порог XP открытия = сумма XP добавленных уроков (XP уже указан в уроках).
- Панель управления: время запуска, продолжительность, расписание
  пуш-уведомлений (анонс/старт/окончание), показ итогов всем.

### Путь пользователя
1. Попап «Поздравляем, доступна еженедельная супергонка» — раз в неделю,
   при первом завершении любого модуля на этой неделе (за ~неделю до гонки).
   Кнопка «Узнать подробнее» → страница гонки.
2. Проходит уроки списка, набирает порог XP → гонка открыта.
3. Прошёл все уроки → остаётся таймер до старта + подписка на пуш о старте.
4. Сб-вс проходит гонку. Прерывание (звонок/батарея) НЕ сжигает попытку:
   прогресс сохраняется, можно продолжить до конца воскресенья; время
   считается только внутри заданий, пауза между ними в зачёт не идёт.

### Подсчёт и итоги
- Один подиум: сортировка по минимуму ошибок, при равенстве — по времени
  (время — только тай-брейкер, чтобы таймер не давил).
- На выходных живой таблицы НЕТ — интрига до понедельника.
- По окончании гонки всем участникам попап: строки мест 1-2-3; если
  пользователь не в тройке — его строка чуть ниже с отступом.
- В итоговой таблице публично только место и очки; свои ошибки и время
  каждый видит только на своём экране результата.

### Достижения (3 шт) и косметика — 1 достижение = 1 вид
| Достижение | Условие | Открывает |
|------------|---------|-----------|
| 10-й уровень | достичь 10-го уровня | подложка под ник |
| Участник гонки | финишировал все задания гонки | рамка вокруг аватара |
| Победитель гонки | топ-3 подиума | медаль у аватара с цифрой 1/2/3 |

Пользователь навешивает открытое в любой комбинации и видит предпросмотр,
как он выглядит в глобальном рейтинге.

### Вкладка «Рейтинг» (новая, в нижнем баре)
- Глобальный рейтинг по XP **за всё время**: строка = аватар (+рамка,
  +медаль), ник (на подложке), уровень.
- Сверху — баннер-вход текущей супергонки (тема, прогресс XP, таймер).
- Субвкладка «Достижения»: список трёх достижений + примерка косметики.

### Технические решения
- Уровни: расширить `xpLevels.js` до 10-го — пороги 5:1600, 6:2400,
  7:3400, 8:4600, 9:6000, 10:8000 XP (названия — предложить на утверждение).
- Ники/аватарки: в `user_profiles` их нет → добавить `nickname`
  (по умолчанию из имени регистрации, редактируется в профиле);
  аватар — кружок с первой буквой и цветом от id (без загрузки фото).
- RLS позволяет читать только свой профиль → для рейтинга нужна публичная
  выборка (view/RPC c ником, XP, уровнем, косметикой — без email и прочего).
- Итоги гонки, медали, попап результатов — серверный пересчёт по окончании
  (по аналогии с pg_cron у пушей).

### Правки по первому тесту (2026-07-10) — реализовано

### Про-модули — супер-урок гонки (решения 2026-07-10, вторая итерация)

- Рейтинг: строки топ-3 в оттенках золото/серебро/бронза; свою строку не

## Золотые билеты + подсказки Финала (решения 2026-07-11)

### Экономика (согласовано)
- **Золотой билет** — топ-валюта. Выдаётся за прохождение Финала модуля
  максимум с **3 подсказками**. Подсказка = ПЕРВОЕ раскрытие перевода
  конкретного сообщения (повторное раскрытие того же — бесплатно).
  Больше трёх — раскрывать не запрещено, но билет в этой попытке не выдаётся.
- **1 модуль = 1 билет навсегда** (реестр `module_tickets`) — редкость
  гарантирована структурно: билетов не больше, чем модулей. Не вышло —
  пересдавай Финал сколько угодно, билет с модуля всё равно единственный.
- **Вход в супергонку = 80% XP подготовки И 1 билет.** Билет списывается
  в попапе-шлагбауме кнопкой «Открыть доступ» (RPC `start_race`, работает
  с анонса до конца гонки), дальше вся гонка без доплат — повторные входы
  бесплатны (`race_ticket_spends`); админ — бесплатно. Кнопка «Начать
  супергонку» зовёт тот же RPC как страховку для путей в обход шлагбаума
  (попап-анонс) — идемпотентно.
- **0 подсказок** = достижение «Чистый финал» (`clean_final`, выдаётся и при
  пересдаче) → открывает **золотую подложку** под ник (`cosmetics.bg2`).
- Верхняя панель (hudBar, ShellV2): уровень · билеты 🎟 · энергия ⚡.
- Баннер гонки в «Рейтинге»: строка «Вход: 1 золотой билет»; тап по баннеру →
  попап-шлагбаум (RaceTicketGate): стоимость, свой баланс, как заработать.
  Кнопка «Открыть доступ» списывает билет и открывает страницу гонки; без
  билета кнопки нет. Вход свободный (кнопка «Войти», билет не тратится):
  админ · гонка завершена · уже финишировал · доступ уже оплачен.
  Попап-анонс недели («Узнать подробнее») открывает страницу напрямую —
  осознанно: там билет спишется при старте супер-урока.
- Анти-чит v1: число подсказок серверу присылает клиент — тот же осознанный
  компромисс, что у времени/ошибок гонки. Баланс билетов живёт только на
  сервере (`user_profiles.tickets`, все изменения через SECURITY DEFINER RPC).

### Реализовано (2026-07-11) — ждёт SQL
Клиент готов: HintBar (3 огонька в Финале), выдача билета после
`complete_lesson`, блок билета в итогах урока, hudBar, вход в гонку через
`start_race`, достижение + подложка в «Кастомизации». **Перед проверкой
применить в Supabase** блок «Золотые билеты + Чистый финал» из
`supabase_schema.sql` — без него `getProfile` упадёт на выборке колонки
`tickets` (профиль/энергия перестанут показываться).

### Звёзды обычных уроков — реализовано (2026-07-11, v3.2.820)
- 1–3 звезды по неверным ответам за прохождение: 3★ — 0 ошибок, 2★ — 1–2,
  1★ — пройден. Ошибки считает свой счётчик плеера (независим от recordStats —
  пересдача «без записи» не дарит 3★). Только уроки между Стартом и Финалом.
- Чистая косметика: на доступ к Финалу не влияют. Хранится ЛУЧШИЙ результат.
- Хранение: сервер `lesson_results.stars` (RPC `save_lesson_stars`, только
  вверх) + localStorage (`lessonStars.js`) — гость и мгновенный показ; схема
  читает максимум из двух.
- Показ: экран итогов урока (StarsBlock) + карточки уроков схемы модуля
  (MgStars вместо «Пройдите и получите» у пройденных).
- **SQL**: блок «Звёзды обычных уроков» в `supabase_schema.sql` (колонка +
  RPC). Без него клиент не ломается: RPC вернёт ошибку → best=0, звёзды
  живут локально.

## Telegram Mini App — план интеграции (2026-07-11, не начато)

Цель: приложение открывается ПРЯМО ВНУТРИ Telegram — пользователь из
телеграм-канала владельца нажимает «Открыть приложение» и попадает в Pithy
без браузера. Технология: Telegram Mini Apps (WebApp) — тот же наш сайт
с Vercel, обёрнутый мини-аппом бота. Отдельного билда НЕ делаем: одна
кодовая база, Telegram определяется на лету.

Каждый этап — самостоятельное ТЗ. Общие правила из CLAUDE.md обязательны
(русский, APP_VERSION при изменении кода, STRUCTURE.md, лимиты строк,
отчёт о строках, пуш только по просьбе).

### Как это устроено (справка для исполнителя)
- Мини-апп привязывается к боту. Ссылка вида `t.me/<имя_бота>/<имя_аппа>`
  публикуется в канале — Telegram сам рисует карточку с кнопкой «Открыть
  приложение». Внутри открывается наш URL (https://pithy-app.vercel.app).
- В мини-аппе доступен объект `window.Telegram.WebApp` (подключается
  скриптом `https://telegram.org/js/telegram-web-app.js`): initData
  (подписанные данные пользователя TG), управление высотой/жестами/кнопкой

- На телефонах мини-апп живёт в webview (не iframe), в Telegram Desktop и

  него использовать CSP `frame-ancestors 'self' https://web.telegram.org`

## Ежедневный стрик + окно наград (решения 2026-07-12, в разработке)

### Суть
Стрик = число дней подряд, когда пользователь ЗАХОДИЛ в приложение (загрузка
приложения, не привязано к прохождению урока). Несколько заходов в один день
считаются одним. За каждый день — начисление в **основной XP** (не отдельная
валюта): 5 XP за обычный день, крупные бонусы за вехи. Получение — **только
вручную**, кнопкой «Забрать награду» в полноэкранном попапе (макет прислан
пользователем — вертикальный путь дней, hero-прогресс до следующей вехи).

### Вехи наград (конфиг в БД, не в коде)
| День | Награда |
|------|---------|
| 1..6, 8..14, 16..29 (не веха) | 5 XP |
| 7 | 150 XP |
| 15 | 300 XP |
| 30 | 500 XP + 1 золотой билет |
| 37, 45, ... | доп. приз (по той же аналогии, точные числа/награды до дня
  182 донабить позже — не блокирует остальную разработку) |
| 182 (полгода) | спецокно + 2 золотых билета |
| 365 (год) | 3 золотых билета |

Хранится в таблице `streak_milestones` (day_number, xp_reward, ticket_reward,
label) — день не из таблицы получает дефолт 5 XP. Редактируется админом прямо
из приложения (вкладка «Админ»), а не правкой SQL — см. этап 8 ниже.

### Заморозки (обе — за золотые билеты)
- **Заморозка** — покупается заранее, **не стакается** (можно иметь только
  одну про запас). Хранится как флаг `has_freeze_charge boolean`. Молча
  списывается сервером в `touch_daily_login`, когда обнаружен пропущенный
  день — спасает стрик без участия пользователя.
- **Авто заморозка** (обычный пользователь) — покупка даёт защиту на **2
  пропущенных дня** (`auto_freeze_charges_left`), повторная покупка
  недоступна, пока пул не обнулится.
- **PRO** — тот же эффект включён навсегда бесплатно: суббота и воскресенье
  всегда прощаются + 1 будний день в неделю прощается автоматически.
- Тап обычного пользователя на кнопку «Авто заморозка» без покупки — шторка
  с объяснением механики + предложение оформить PRO.

### Показ окна наград
- Полноэкранный попап (не блок в профиле) — **раз в день, после первого
  пройденного урока за этот день**.
- Ручной вход — блок «Ежедневные награды» в профиле, открывает тот же попап
  в любой момент.
- В рейтинге — маленький значок 🔥 с числом дней рядом с ником (виден всем).

### Защита от накрутки
Всё серверное: баланс XP/билетов/заморозок живёт только в `user_profiles`,
все изменения — через SECURITY DEFINER RPC (как `spend_energy`/`complete_lesson`),
клиент не передаёт суммы. `streak_milestones` — RLS `select` всем,
`insert/update/delete` только `is_admin()` (тот же паттерн, что у `curricula`/
`races`) — обычный пользователь не может изменить награды даже прямым
запросом в обход интерфейса.

### Этапы разработки
1. БД: колонки `user_profiles` + таблица `streak_milestones` + RPC
   (`touch_daily_login`, `claim_streak_reward`, `buy_streak_freeze`,
   `buy_auto_freeze`) + `get_leaderboard` с `current_streak`.
2. Клиент: `streakApi.js` + вызов `touchDailyLogin()` при загрузке приложения.
3. Рейтинг: значок 🔥N у ника в топе.
4. Профиль: блок «Ежедневные награды» (статус + кнопка попапа).
5. Попап наград по макету: hero-прогресс, путь дней, «Забрать награду».
6. Автопоказ попапа раз в день после первого урока.

### Стрик v2 — пояса, автоклейм, «Забрать всё», пуш вехи (решения 2026-07-16)
Реализовано (SQL-блок «Стрик v2» в supabase_schema.sql + клиент + push-trigger):
- **Часовой пояс устройства**: граница суток стрика — пояс телефона
  (`Intl.DateTimeFormat().resolvedOptions().timeZone` → `user_profiles.tz`),
  VPN не влияет (пояс из настроек ОС, не из IP). Невалидный пояс от клиента
  отбрасывается сервером.
- **Анти-накрутка**: новый день серии засчитывается не чаще, чем раз в 12
  реальных часов (`last_streak_increment_at`); «слишком ранний» заход
  помечает день посещённым (серия не сгорает), +1 придёт со следующим днём.
  Сдвиг пояса назад (отрицательный gap) серию не роняет.
- **Автоклейм**: при сбросе серии незабранные награды не сгорают — сервер
  начисляет их сам; клиент показывает плашку «Серия N дн. прервалась,
  начислено +X XP» в окне наград (localStorage `pithy_streak_reset_info`).
- **«Забрать всё»**: RPC `claim_streak_rewards_all()` начисляет все
  накопленные дни одной операцией; кнопка в попапе при 2+ незабранных днях.
- **Пуш «накануне вехи»** (`streak_milestone_eve`): вечером тем, кто сегодня
  уже заходил и у кого завтра день-веха — «Завтра 7-й день, ждёт 150 XP».
  Плейсхолдеры шаблона: {streak} {day} {xp} {tickets}.
- **Почасовой вечерний cron**: `push_audience_evening()` сама выбирает тех,
  у кого в их поясе сейчас 19:xx; дедупликация — по локальной дате.
- Цены заморозок остаются в SQL (решение: в админку не выносить).

## Код-сплиттинг бандла — план реализации (2026-07-11, не начато)

### Зачем (что даст)
- Сейчас весь JS — один файл `dist/assets/index-*.js` на **765 КБ** (gzip 222);
  Vite ругается предупреждением при каждой сборке. Обычный пользователь
  скачивает и парсит код canvas-редактора и админки (~4 200 строк —
  `src/features/canvas/` + `src/features/admin/`), которые видит только админ.
- После разбивки: основной бандл ожидаемо −20–35%, предупреждение Vite
  уходит, первый вход (особенно на телефоне по 3G) быстрее — меньше скачать
  и меньше распарсить. Точные цифры замерить до/после по выводу `npm run build`.

### Почему это безопасно именно здесь
`AdminV2` и `CanvasPage` статически импортируются РОВНО в одном месте —
`src/app/ShellV2.jsx` (проверено grep'ом). Больше никто из основного кода
их не тянет, значит `React.lazy` отделит их в отдельные chunk'и чисто.
Вкладка «Админ» рендерится только при `isAdmin`, canvas-оверлей — только
когда `canvasLesson != null`: у обычного пользователя эти chunk'и даже
не начнут скачиваться.

### Шаги

1. **Хелпер перезагрузки** — новый файл `src/shared/lib/lazyRetry.js`
   (~15 строк). Известная беда Vite: после деплоя старые chunk-хэши отдают
   404, и `import()` у пользователей с открытой вкладкой падает. Обёртка:
   один раз перезагрузить страницу при ошибке загрузки chunk'а
   (флаг в sessionStorage, чтобы не зациклиться):
   ```js
   export function lazyRetry(importFn, key) {
     return importFn().catch(err => {
       const k = `pithy_lazy_reload_${key}`
       if (!sessionStorage.getItem(k)) {
         sessionStorage.setItem(k, '1')
         window.location.reload()
       }
       throw err
     })
   }
   ```
2. **ShellV2.jsx**: заменить два статических импорта на lazy:
   ```js
   import { lazy, Suspense } from 'react'
   const AdminV2   = lazy(() => lazyRetry(() => import('../features/admin/AdminV2.jsx'), 'admin'))
   const CanvasPage = lazy(() => lazyRetry(() => import('../features/canvas/CanvasPage.jsx'), 'canvas'))
   ```
   Обернуть места рендера в `<Suspense fallback={...}>`:
   - вкладка Админ: fallback `<div className="shellV2Panel">Загрузка…</div>`
     (в стиле существующих заглушек, НЕ белый экран);
   - canvas-оверлей: fallback — тот же `.shellV2CanvasOverlay` с текстом
     «Загрузка редактора…», чтобы оверлей не мигал.
3. **Ничего больше не трогать.** Плеер, лента, профиль, рейтинг, гонка —
   пользовательское ядро, остаются в основном бандле (все вкладки
   смонтированы всегда — ленивость сломала бы прогрев ленты). CSS не
   разбивать: `index.css` глобальный, это отдельная задача, не в этом заходе.
4. **Проверка сборки**: `npm run build` → в `dist/assets` появились
   отдельные `AdminV2-*.js` и `CanvasPage-*.js`; основной `index-*.js`
   заметно похудел; предупреждение «chunks larger than 500 kB» исчезло
   (если нет — найти, кто ещё статически импортирует canvas/admin:
   `grep -r "features/canvas\|features/admin" src --include=*.jsx -l`).
5. **Проверка в превью** (dev-сервер `pithy-dev`):
   - гость: лента открывается, во вкладке Network НЕТ chunk'ов admin/canvas;
   - админ: вкладка «Админ» открывается (после короткого fallback),
     ✎ на схеме модуля открывает canvas-редактор; консоль без ошибок.
6. **Правила проекта**: поднять `APP_VERSION`; новый файл `lazyRetry.js`
   добавить строкой в `STRUCTURE.md`; в конце ответа — список файлов с
   итоговым числом строк; отвечать по-русски.

### Критерий готовности
Основной `index-*.js` < 500 КБ (предупреждение Vite исчезло), админ-функции
работают как раньше, гость не скачивает админ-код (видно в Network).
