# Pithy App ver 2.0 — задача

## Контекст

Старый проект (`C:\PITH\Development\Development\pith-app_folder\pith-app`) — рабочее приложение
с конструктором уроков (чат-сценарии), но серверная часть и логика синхронизации файлов
обросли багами за много итераций (см. его `SYNC_PLAN.md` и историю правок C342–C419 в его `CLAUDE.md`).

Здесь, в ver 2.0, серверная часть строится заново, но не с нуля — взяты только рабочие,
самодостаточные куски старого проекта, которые отвечают за хранение уроков и файлов:

| Файл | Назначение |
|------|------------|
| `supabase_schema.sql` | Схема БД: таблица `lessons` (+ `user_profiles`, `lesson_results` — пока не нужны для админ-панели, но оставлены для справки) |
| `supabase/config.toml` | Конфиг Supabase CLI (project_id, локальные порты) |
| `supabase/functions/r2-upload-url/` | Edge function: генерирует presigned PUT URL для прямой загрузки файла в Cloudflare R2 (бакет `pithy-files`) |
| `supabase/functions/r2-delete/` | Edge function: удаляет файл из R2 по его публичному URL |
| `src/shared/api/supabase.js` | Инициализация клиента Supabase JS v2 (читает `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY`) |
| `src/shared/lib/r2.js` | `uploadToR2(file)` — получает presigned URL у edge function и грузит файл через `XMLHttpRequest` (не `fetch` — на больших файлах Chrome рвёт соединение, `ERR_CONNECTION_RESET`); `deleteFromR2(url)` — вызывает `r2-delete` |
| `src/shared/lib/localFileStore.js` | Обёртка над IndexedDB (`lfSave/lfGet/lfGetAll/lfDelete`) — локальный кэш файлов до синхронизации, не упирается в лимит `localStorage` (~5MB) |
| `.env.local` | Реальные ключи Supabase (скопированы из старого проекта) |

**Backend переиспользуется** — тот же Supabase-проект и тот же R2-бакет `pithy-files`, что и в
старом приложении (решение пользователя: не поднимать новый, чтобы не терять время сейчас).
Из старого проекта НЕ копировались: React-компоненты, `BlockEditorChat.jsx` и вся логика
чат-конструктора, mock-слой `adminLessons.js` — это и есть тот самый "кучу багов" код, который
переписывается заново.

**Важный пробел**: в текущей схеме (`supabase_schema.sql`) нет отдельной таблицы для общего
реестра файлов — таблица `lessons` хранит медиа-ссылки только внутри `blocks` (jsonb). Для
админ-панели, описанной ниже, нужна отдельная таблица (например `files`: имя, размер, R2 URL,
дата загрузки, статус) — её ещё нет, создание входит в следующий этап.

## Задача: примитивная админ-панель (этап 1)

Это самый первый этап разработки приложения. Цель этапа — добиться **стабильной** загрузки
файлов на сервер и их получения пользователем как на ПК, так и на смартфоне. Никакого
конструктора уроков на этом этапе нет — только файлы.

Два таба: **Пользователь** и **Админ**.

### Таб «Пользователь»
- Видит только файлы, которые есть на сервере.
- Может воспроизвести: фото (просмотр), видео (воспроизведение), аудио (воспроизведение).

### Таб «Админ»
- Создаёт строку → выбирает файл с ПК.
- В строке видно: выбранный файл, вес файла (столбец), статус (на сервере / нет), время
  загрузки на сервер.
- Может удалить файл с сервера прямо из строки.
- **Все изменения применяются только после нажатия главной кнопки «Синхронизировать с
  сервером»** — выбор файла и удаление до этого момента — локальное, неподтверждённое действие.
- Кнопка «Активировать дебаг» — включает подробные логи в консоли: как файл уходит от админки,
  как файл загружается у пользователя, время загрузки и т.д.

## Что обсудили / решили
- Backend: переиспользуем существующий Supabase-проект и R2-бакет `pithy-files` (не новый).
- Стек: React + Vite + Supabase JS v2 — та же связка, что в старом проекте (вытащенные
  `r2.js`/`localFileStore.js`/`supabase.js` — чистый JS, переиспользуются как есть).
- Нужна новая таблица `files` в Supabase — отдельный реестр загруженных файлов (имя, размер,
  R2 URL, дата загрузки, статус), не привязанный к урокам. Список для таба «Админ»/«Пользователь»
  строится из неё, а не из перечисления объектов R2 напрямую.
- Авторизация таба «Админ»: простой общий пароль (без полноценного Supabase Auth на этом этапе).
- Проверка получения файлов на смартфоне: через деплой на Vercel (стабильный HTTPS URL), а не
  через dev-сервер по локальной Wi-Fi сети.

## Открытые вопросы (обсуждаем до начала кодинга)
- Лимиты: какие типы файлов и какой максимальный размер поддерживаем — пока без лимита,
  отложено по просьбе пользователя.
- Пароль админа хранится в `VITE_ADMIN_PASSWORD` (клиентский env var) — простая защита от
  случайного доступа, не криптостойкая (видна в собранном бандле). Решение принято, можно
  заменить на Supabase Auth позже.
