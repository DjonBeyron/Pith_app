# Pithy App ver 2.0 — инструкции для Claude Code

Этот файл подгружается автоматически в каждую новую сессию в этой папке. Следуй ему всегда.

## Язык
Отвечай на русском, коротко и по делу.

## Три файла — три разные роли (не путать)
- **`CLAUDE.md`** (этот файл) — правила «как писать код здесь». Не хранит список файлов и не
  хранит историю задачи — только правила.
- **`PROJECT.md`** — что строим, какие решения уже приняты, какие вопросы открыты. Обновляется,
  когда меняется план или принимается решение.
- **`STRUCTURE.md`** — построчная карта всех файлов и папок: что лежит где и зачем. Обновляется
  при каждом добавлении/удалении файла.

## Главное правило: маленькие файлы, чёткое разделение
В старом проекте (`pith-app_folder`) один файл (`BlockEditorChat.jsx`) дорос до тысяч строк за
сотни мелких правок — стало невозможно с ним работать. Здесь это не повторяем.

- **Мягкий ориентир — 250 строк на файл.** Если файл растёт за эту границу — это сигнал
  разделить его по смыслу (не дожидаясь твёрдого лимита).
- **Твёрдый потолок — 400 строк.** Это проверяется автоматически: `npm run lint` падает с
  ошибкой `max-lines`, если файл больше 400 строк (см. `eslint.config.js`). Запускай `npm run
  lint` после крупных изменений.
- **Один файл — одна ответственность.** Если файл делает две разные вещи, которые можно
  разделить (например, форма пароля и таблица файлов) — раздели на два файла, даже если оба
  короткие.
- Когда файл/папка делает что-то новое и для этого создаётся отдельный модуль — выноси в
  подпапку, а не дописывай в существующий большой файл.

## Структура папок (`src/`)
- `src/app/` — сборка приложения целиком (например `App.jsx` — переключение вкладок). Сюда
  идёт только то, что связывает крупные части вместе, без бизнес-логики внутри.
- `src/features/<имя>/` — одна вкладка/функция приложения = одна папка (сейчас: `admin/`,
  `user/`). Если фича разрастается — внутри её папки создавай несколько файлов
  (`AdminTab.jsx`, `AdminFileRow.jsx`, `useAdminSync.js`...), а не один большой файл.
- `src/shared/api/` — всё что говорит напрямую с Supabase (клиент, авторизация).
- `src/shared/lib/` — вспомогательные функции без интерфейса (работа с R2, IndexedDB, debug-лог
  и т.п.) — переиспользуются разными фичами.
- `src/shared/ui/` — кусочки интерфейса, общие для нескольких фич (кнопка, карточка и т.п.).
  Создавай файл здесь только когда кусок интерфейса реально нужен в двух местах — не раньше.

## Обязательное правило: обновляй `STRUCTURE.md`
Каждый раз, когда создаёшь новый файл — в том же шаге добавь одну строку в `STRUCTURE.md`:
что это за файл и зачем он нужен, простыми словами. Удалил файл — убери его строку. Это не
опционально: `STRUCTURE.md` должен оставаться точным отражением реального дерева файлов после
каждой сессии.

## Стиль кода
- Имена файлов компонентов — `PascalCase.jsx` (один компонент = один экспорт по умолчанию).
- Имена файлов с обычными функциями — `camelCase.js`.
- Без лишних абстракций: не вводи слой/обёртку, пока не нужна минимум двум местам в коде.

## Git
Пушить в git только по явной просьбе пользователя. После каждого пуша правило сбрасывается —
в следующий раз нужна новая явная просьба.

## Стек
React 19 + Vite 8, Supabase JS v2 (база данных + edge functions), Cloudflare R2 (файлы).
Подробности по бэкенду — в `PROJECT.md` и `supabase_schema.sql`.
