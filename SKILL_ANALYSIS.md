# Система анализа знаний (приоритеты уроков)

Что это: стартовый урок незаметно диагностирует пользователя, и список уроков
превращается в персональный маршрут — каждый урок получает полоску приоритета.
Пользователь не решает сам, что учить — система подсвечивает пробелы.

---

## 1. Термины

| Термин | Что значит |
|--------|-----------|
| **Урок-источник** | Урок, где пользователь отвечает (стартовый урок с разбором фразы) |
| **Урок-цель** | Урок, которому уходят данные ответа (урок про слово: «i'm», «to»...) |
| **Событие** | Одна запись о действии пользователя (ответил верно/неверно, пропустил...) |
| **Сессия** | Одно прохождение урока-источника; при пересдаче события старой сессии заменяются новой |
| **Сигнал** | Смысл ответа-самооценки: «знает» или «не знает» (для вариантов без верного ответа) |

---

## 2. Что видит пользователь

### Полоски приоритета (левый край карточки урока)

| Полоска | Приоритет | Текст | Когда |
|---------|-----------|-------|-------|
| 🟠 оранжевая | **Высокий** | «Наиболее важные для вас уроки» | были ошибки без последующего верного ответа |
| 🟡 жёлтая | **Средний** | «Полезные уроки для развития» | ответил верно, но не сразу / долго думал / просил объяснение |
| 🔵 синяя | **Низкий** | «Можно изучить позже» | всё верно с первой попытки — тема знакома |
| — без полоски | нет данных | — | по этому уроку не было ни одного вопроса |

### Попап-легенда «Приоритеты уроков»

Отдельное окно поверх схемы уроков (схема затемнена), закрывается **только вручную**.
Содержимое — легенда трёх приоритетов (иконка + название + описание, как на макете).

Когда показывается:
1. после **первого** прохождения стартового урока (когда полоски появились впервые);
2. после повторного прохождения, если пользователь выбрал режим **«с обновлением анализа»**.

Когда НЕ показывается: обычное открытие вкладки «Уроки», повтор «без записи».

### Диалог перед повторным прохождением

Если урок уже пройден и содержит привязанные ответы — перед стартом окно с выбором:

```
🔄 Пройти с обновлением анализа   — ответы перезапишут карту знаний,
                                     статусы уроков пересчитаются
👁 Пройти без записи               — просто повторить, анализ не изменится
    [ Отмена ]
```

Первое прохождение диалог не показывает.

## 3. Как настраивает админ

Принцип один: **куда смотрит ответ — туда уходят данные**. Никаких новых триггеров,
триггеры остаются только для управления потоком чата.

### Ноды «Собрать фразу» и «Выбрать фото» — привязка на уровне ноды

Ответ один, поэтому один дропдаун на ноду:

```
Нода «Собери: I'm trying to cook»
  → Урок: [ trying ▾ ]        ← одно поле, всё
```

### Нода «Выбор слова» — привязка на уровне варианта

Два типа вопросов, настраиваются по-разному:

**Тип 1 — проверка знания** (есть ✓ на верном варианте). Достаточно указать урок
на ноде — сигнал correct/wrong выводится из галочки ✓ автоматически:

```
Нода → Урок: [ i'm ▾ ]
  ○ I trying to sleep          (выбрал → wrong уроку i'm)
  ● I'm trying to sleep  ✓     (выбрал → correct уроку i'm)
  ○ Im trying to sleep         (выбрал → wrong уроку i'm)
```

**Тип 2 — самооценка** (верного ответа нет). У каждого варианта свои поля
«→ Урок» + «Сигнал», варианты могут ссылаться на РАЗНЫЕ уроки:

```
  ○ Расскажи про i'm     → урок: i'm,    сигнал: «не знает»
  ○ Что такое please?    → урок: please, сигнал: «не знает»
  ○ Конечно знаю         → урок: please, сигнал: «знает»
  ○ Промолчать           → без привязки (сюжетный, данных не даёт)
```

Вариант без привязки — просто сюжетный, в анализ ничего не пишет.

## 4. Что записывается (событие)

Сущность — **не баллы, а лог сырых фактов**. Ничего не суммируется при записи;
статус каждый раз вычисляется заново по правилам. Формулу можно менять,
не трогая собранные данные.

```json
{
  "lessonId":  "урок-to",          // урок-цель (кому данные)
  "type":      "wrong",            // correct | wrong | know | dont_know
  "attempt":   2,                  // номер попытки внутри ноды
  "timeMs":    6300,               // от появления панели до первого клика
  "option":    "I will try please everyone",  // что именно выбрал/собрал
  "sessionId": "s-2026-07-03-1",   // прохождение урока-источника
  "at":        "2026-07-03T14:20"  // когда
}
```

Правила записи:
- **Таймер** стартует, когда панель (выбор слова / сборка / фото) закончила
  анимацию и стала кликабельной. Аудио учителя и «печатает…» в замер не входят.
- **Попытки** складываются сами: если «попробуй ещё раз» ведёт на вторую ноду,
  привязанную к тому же уроку, — это попытка 2 в той же копилке.
- **Пересдача «с обновлением»**: события новой сессии ЗАМЕНЯЮТ старые события
  этого урока-источника в расчёте. Старые сессии остаются в логе как история.
- Принцип: **записываем щедро, судим скупо** — поле `option` и `at` пишем сразу,
  даже если правила их пока не используют (пригодятся позже без миграций).

## 5. Как считается приоритет

Каскад правил сверху вниз, первое сработавшее — ответ. Берутся события
**последней сессии** каждого урока-источника.

```
1. Событий нет                          → без полоски (нет данных)
2. Есть wrong БЕЗ последующего correct  → 🟠 ВЫСОКИЙ
3. Есть correct, но:
     · со 2-й и более попытки, ИЛИ
     · дольше THRESHOLD_SLOW, ИЛИ
     · был сигнал «не знает»            → 🟡 СРЕДНИЙ
4. Всё correct с 1-й попытки            → 🔵 НИЗКИЙ
5. Только «знает» (без фактических
   ответов)                             → 🔵 НИЗКИЙ (слабый: первая же
                                           ошибка позже перекрасит)
```

Два встроенных приоритета вместо весов:
- **Факт сильнее самооценки.** «Конечно знаю» + потом ошибка = 🟠, самооценка отброшена.
- **Свежее сильнее старого.** `wrong → wrong → correct` = 🟡 (в итоге разобрался),
  а `correct → wrong` = 🟠 (знание не удержалось).

Время — вторичный уточнитель: оно **никогда не даёт 🟠**, только сдвигает 🔵 → 🟡.
Долгий ответ ≠ незнание (мог отвлечься), а ошибка — это ошибка.

### Несколько событий на один урок — это норма, не конфликт

В копилку урока может ссылаться сколько угодно ответов из разных модулей.
Каскад смотрит на весь набор с учётом порядка:

| События (по порядку) | Итог | Почему |
|---|---|---|
| `dont_know` → `correct` | 🟡 | «расскажи про i'm» + верная сборка = только что узнал, закрепить |
| `know` → `correct` | 🔵 | знание заявлено и подтверждено фактом |
| `know` → `wrong` | 🟠 | факт сильнее самооценки |
| `dont_know` → `wrong` → `correct` поп.2 | 🟡 | в итоге разобрался |
| `correct` → `correct` → `wrong` | 🟠 | свежее сильнее: знание не удержалось |

Именно поэтому сущность — лог, а не баллы: одно суммарное число не отличило бы
«не знал, но научился» от «наполовину знает», а последовательность событий — отличает.

Пороги — константы в одном файле (`skillScore.js`):

```
THRESHOLD_SLOW = 15000   // мс; «верно, но долго думал»
ATTEMPTS_OK    = 1       // с какой попытки ответ ещё «чистый»
```

### Пример сквозного расчёта

Пользователь прошёл стартовый урок «I'm trying to please both»:

```
Действие                                   → Событие
──────────────────────────────────────────────────────────────
«I'm trying to sleep» верно, сразу, 4 с    → i'm:    correct, поп.1
про please нажал «Конечно знаю»            → please: know
собрал «I'm trying to cook» со 2-й попытки → trying: correct, поп.2
фразу с will собрал неверно                → to:     wrong
нажал «А тут какое-то to?»                 → to:     dont_know
после объяснения — снова неверно           → to:     wrong, поп.2
фото для focus выбрал верно сразу          → trying: correct, поп.1

Расчёт:
i'm     [correct с 1-й]                    → 🔵 Низкий
trying  [correct поп.2] + [correct поп.1]  → 🟡 Средний
to      [wrong, dont_know, wrong]          → 🟠 Высокий
please  [know без фактов]                  → 🔵 Низкий (слабый)
both    [нет событий]                      → без полоски
```

Урок **to** — доказательство работы системы: три независимых события из разных
модулей (сборка фразы + выбор слова) сошлись в одной копилке, потому что каждый
ответ нёс адрес доставки.

## 6. Хранение

| Кто | Где |
|-----|-----|
| Залогиненный | Supabase: колонка `answers jsonb` в существующей `lesson_results` (пишется в конце урока вместе с XP) |
| Гость | `localStorage` (по аналогии с локальным XP в `localProfile.js`) |

Серверной логики нет — расчёт приоритетов на клиенте при открытии вкладки «Уроки»,
как и весь плеер.

## 7. План реализации

Детальный поэтапный план с тестами каждого этапа — в `SKILL_ANALYSIS_PLAN.md`.
Ниже — краткая сводка.

| Шаг | Что делаем | Файлы (оценка) |
|-----|-----------|----------------|
| 1 | Схема событий + расчёт приоритетов + пороги | `src/shared/lib/skillScore.js` (~150 строк) |
| 2 | Настройка в редакторе: дропдаун «→ Урок» на ноде (фраза/фото), поля «→ Урок»+«Сигнал» у вариантов выбора слова | правки `NodeWordChoicePicker.jsx` + общий мини-компонент дропдауна (~100 строк) |
| 3 | Сбор событий в плеере: таймер панели, цепляемся к существующим `handleWordAnswer`/`handlePhraseAnswer`/`handlePhotoPick`, флаг `recordStats` | хук `useAnswerStats.js` (~80 строк) |
| 4 | Сохранение: `lesson_results.answers` (SQL) + localStorage для гостя | `supabase_schema.sql`, `skillStats`-хранилка |
| 5 | Полоски приоритета из расчёта вместо статичного `PRIORITY` | правка `ModuleGraph.jsx` (~20 строк) |
| 6 | Попап-легенда «Приоритеты уроков» (оверлей, затемнение, ручное закрытие) | новый компонент + css (~120 строк) |
| 7 | Диалог пересдачи («с обновлением» / «без записи») | новый компонент (~80 строк) |
